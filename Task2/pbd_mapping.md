# Маппинг 7 принципов Privacy by Design на блоки целевой архитектуры

Этот документ показывает, **как каждый из семи принципов Privacy by Design**
(Ann Cavoukian) реализуется в целевой архитектуре «Медикаменте». Это и есть ответ
на вопрос задания «предложить новые блоки в C4, которые обеспечат соблюдение
принципов PbD во многих блоках реализуемой системы».

## Сводная таблица

| # | Принцип PbD | Архитектурные блоки | Конкретные механизмы |
|---|-------------|--------------------|-----------------------|
| 1 | **Proactive not reactive; Preventative not remedial** | DLP, Data Catalog, SIEM, CI/CD Privacy Gates | Скан-агенты тегируют новые поля автоматически, релизы блокируются при обнаружении обработки L4 без OPA-политики, регулярные PIA |
| 2 | **Privacy as the Default Setting** | API Gateway, OPA, Catalog, KMS | По умолчанию ничего не возвращается без явной политики; новые поля без тегов считаются `L3`; все БД создаются с включённым TDE |
| 3 | **Privacy Embedded into Design** | KMS, Tokenization, Catalog, OPA, Consent | PbD-блоки — это **обязательная платформа**, сервисы не могут писать в БД минуя KMS. Шаблоны микросервисов уже подключают клиенты этих блоков |
| 4 | **Full Functionality – Positive-Sum, not Zero-Sum** | Анонимизированные витрины, JIT-доступ через DPO, ЭП через ЕСИА | Бизнес-функции работают полностью: запись, оплата, аналитика. PbD не ломает UX, а аналитика остаётся возможной — на обезличенных данных |
| 5 | **End-to-End Security – Lifecycle Protection** | TLS/mTLS, KMS, TDE, Field-level encryption, Retention Engine, Backup Encryption | Шифрование: in-transit (TLS 1.3), at-rest (TDE), at-use (field-level + OPA), при удалении (Retention Engine с ключами crypto-shredding) |
| 6 | **Visibility and Transparency** | SIEM, DPO Console, Data Catalog, Lineage (OpenLineage), Consent Service | DPO видит, что и кому известно; пациент в ЛК видит, какие согласия он подписал и какие данные о нём обрабатываются; lineage позволяет ответить на запрос «где мои данные» |
| 7 | **Respect for User Privacy – Keep it User-Centric** | Consent Management Service, Self-Service UI, Retention Engine, ЕСИА | Пациент может: подписать/отозвать согласие с ЭП, скачать свои данные, запросить удаление, увидеть журнал доступа к своим данным |

## Детально по каждому принципу

### 1. Proactive not Reactive

**Что значит:** риски конфиденциальности обнаруживаются и закрываются **до** того,
как они материализуются.

**Что в архитектуре:**

- **DLP в CI/CD**. Пайплайн релиза падает, если контракт API возвращает поле,
  помеченное `sensitivity:l4`, без OPA-политики и без причины.
- **Авто-классификатор Data Catalog**. Новые БД и таблицы автоматически
  сканируются (regex + ML) — найденные ПДн тегируются.
- **PIA (Privacy Impact Assessment) шаблон**. Любая новая фича с обработкой L3+
  обязана пройти PIA в Confluence/Notion с подписью DPO.
- **Threat modeling**. STRIDE/LINDDUN на этапе дизайна.

### 2. Privacy as the Default

**Что значит:** если разработчик ничего специально не сделал — данные защищены.

**Что в архитектуре:**

- **OPA по умолчанию `deny`**. API GW не пропускает запрос, если нет явной
  разрешающей политики.
- **БД создаются с включённым TDE**. Шаблон Terraform не позволяет создать
  PostgreSQL без TDE.
- **Логи маскируются по умолчанию**. Logback-stack автоматически redact-ит поля с
  тегом `protect:mask-in-logs`.
- **Новые поля без тегов считаются `sensitivity:l3`**. Это «безопасный default» —
  пока не доказано обратное, поле считается конфиденциальным.

### 3. Privacy Embedded into Design

**Что значит:** privacy — это не плагин, а часть фундамента архитектуры.

**Что в архитектуре:**

- **Privacy & Security Platform — отдельный boundary в C4-Container**. Это не
  «приложение для безопасности», а **общая платформа**, которой пользуются все
  бизнес-сервисы.
- **Скелетный шаблон микросервиса** (`spring-boot-archetype`) уже включает
  клиенты KMS, Tokenization, Consent, OPA, Catalog. Разработчик не может
  «забыть» их подключить.
- **gRPC-контракты PbD-сервисов** покрыты регрессионными тестами в каждом
  сервисе.

### 4. Full Functionality — Positive-Sum

**Что значит:** privacy не противопоставлена бизнес-функциям. И функция, и
privacy достигаются одновременно.

**Что в архитектуре:**

- **Аналитика работает полноценно** — но не на raw, а на анонимизированных
  данных. Бизнес-KPI считаются корректно.
- **Лаборатория получает токены** — медицинский смысл взаимодействия не
  страдает.
- **Платёж проходит как обычно** — но PAN живёт только в токенайзере.
- **Запись через моб. приложение** удобна для пациента, и при этом данные
  шифруются.

### 5. End-to-End Security

**Что значит:** защита данных охватывает весь жизненный цикл — от сбора до
удаления.

**Что в архитектуре:**

| Состояние данных | Защита |
|------------------|--------|
| In-transit (наружу) | TLS 1.3, mTLS, JWS подпись |
| In-transit (внутри) | mTLS между сервисами в кластере (service mesh / istio) |
| At-rest (БД) | PostgreSQL TDE + pgcrypto field-level для L4+ |
| At-rest (объекты) | S3 SSE-KMS, per-tenant ключи |
| At-rest (бэкапы) | Отдельный backup-ключ KMS, off-site |
| At-use (в памяти) | OPA-проверки, маскирование в UI, минимальный TTL data-key |
| At-deletion | Crypto-shredding (удаление ключа = безвозвратное удаление данных) |
| В аналитике | Анонимизация + дифф. приватность |

### 6. Visibility and Transparency

**Что значит:** все стороны (пациент, DPO, регулятор) могут увидеть, как
обрабатываются данные.

**Что в архитектуре:**

- **SIEM (Elastic + Wazuh)** — единый журнал событий, ретеншн 3 года.
- **DPO Console** — реестр согласий, обращений субъектов, инцидентов.
- **Data Catalog с lineage (OpenLineage)** — ответ на вопрос «где живут
  мои данные» в виде графа.
- **Пациент в ЛК** видит:
  - какие согласия он подписал и когда (с возможностью отозвать);
  - какие данные о нём хранятся (по категориям);
  - кто и когда обращался к его медкарте (упрощённый журнал доступа);
  - кому переданы его данные (лаборатория и т.п.).
- **Публичная политика обработки ПДн** в открытом доступе.

### 7. Respect for User Privacy

**Что значит:** интерфейсы и процессы поставлены на сторону пользователя.

**Что в архитектуре:**

- **Consent Management Service** + **ЕСИА** для юридически значимой ЭП.
  Согласие — атомарное, разделённое по целям (обработка анализов, маркетинг,
  передача в лабораторию и пр.). Можно отозвать каждое отдельно.
- **Self-Service «Удалить мои данные»**. Запрос порождает событие
  `subject.erasure_requested` → Retention Engine выполняет каскадное
  удаление/обезличивание во всех доменах.
- **Self-Service «Скачать мои данные»** (право на переносимость).
- **Default-минимальная анкета**. Расширенные поля собираются только при
  явном указании цели.
- **Уведомления об инцидентах**. Если по данным субъекта произошёл инцидент,
  система автоматически готовит уведомление по 152-ФЗ.

## Сводный итог: что нового добавилось относительно As-Is

| Категория | Было (As-Is) | Стало (To-Be, с PbD-блоками) |
|-----------|--------------|------------------------------|
| Управление ключами | Отсутствует | KMS / Vault, ротация, crypto-shredding |
| Шифрование БД | Нет | TDE + pgcrypto field-level |
| Канал | Открытый (e-mail, OLE) | TLS 1.3 / mTLS / JWS |
| Доступ | AD-группы «всё/ничего» | API GW + OPA по тегам и контексту |
| Согласия | Бумажные | Consent Service + ЕСИА + отзыв в ЛК |
| Право быть забытым | Невыполнимо | Retention Engine + subject.erased |
| Аудит | Нет | SIEM (Elastic + Wazuh), 3 года |
| Тегирование | Нет | Data Catalog (Atlas / OpenMetadata) |
| Аналитика | На сырых Excel | Anonymization Pipeline + анонимизированные витрины |
| Sensitive (L4+) | Лежит вместе со всем | Отдельная БД с отдельным KMS-ключом + ABAC |
| Документы | Сканы на общем диске | S3 SSE-KMS + теги + retention |
| Платёжные карты | Видны в Excel | Vault Transform Tokenization |
| Прозрачность для пациента | Нет | Личный кабинет с реестром согласий, доступом к данным, журналом доступа |

Эти блоки и их связи и есть **«новые блоки в архитектуру C4, обеспечивающие
соблюдение принципов Privacy by Design во многих блоках реализуемой системы»**,
как требует Задание 2.
