# Task 2 — Целевая архитектура (C4) с поддержкой Privacy by Design

В этой директории описана **To-Be архитектура системы «Медикаменте» MVP** с учётом
семи принципов Privacy by Design (PbD) и аналитического слоя, работающего по тем же
принципам.

## Краткое резюме архитектурных решений

В целевой архитектуре добавляются следующие **специализированные блоки Privacy by
Design**, которых не было в As-Is:

| Блок | Роль |
|------|------|
| **KMS / Vault** | Управление ключами шифрования (envelope encryption) |
| **PII Tokenization Service** | Замена ПДн и платёжных данных на токены |
| **Data Catalog (Apache Atlas / OpenMetadata)** | Каталог данных + теги чувствительности (см. `Task1/tagging.md`) |
| **Consent Management Service** | Согласия субъектов и право отозвать обработку |
| **Policy Decision Point (OPA / Cedar)** | Применение политик ABAC по тегам |
| **API Gateway с PbD-плагинами** | Точка входа: TLS, JWT, OPA-проверки, лимиты |
| **Audit / SIEM (Elastic + Wazuh)** | Журналирование доступа, алерты |
| **DLP (InfoWatch / Kaspersky)** | Контроль исходящих данных |
| **Retention Engine** | Автоматическое удаление по срокам и запросам |
| **Anonymization Pipeline (Spark / ARX)** | Слой обезличивания данных для аналитики |
| **Privacy-aware Data Lake + Mart** | Озеро + витрины, не содержащие L4+ данных |

Эти блоки **встроены в каждый поток** и применяются автоматически — это и есть
реализация принципа «Privacy as default» и «Privacy embedded into design».
