# Аудит безопасности данных «Медикаменте» — проблемные зоны (As-Is)

Документ содержит сводный перечень проблем, выявленных в ходе аудита процессов
компании «Медикаменте» и сопоставления их с требованиями российского законодательства
и архитектурных практик в области защиты конфиденциальных данных.

## Нормативная база, по которой выполнялось сопоставление

| Норма | Что регулирует |
|-------|----------------|
| ФЗ-152 «О персональных данных» | Согласия, обработка, передача, сроки хранения, локализация ПДн |
| 323-ФЗ «Об основах охраны здоровья граждан» | Врачебная тайна, доступ к медкарте |
| Приказ ФСТЭК России № 21 | Технические меры защиты ПДн в ИСПДн |
| ПП РФ № 1119 | Уровни защищённости ИСПДн (УЗ-1…УЗ-4) |
| ГОСТ Р 57580 | Защита информации финансовых организаций (применяется частично, кассы) |
| Privacy by Design (7 принципов) | Архитектурные практики |
| OWASP / NIST SP 800-53 | Контроли безопасности данных |

> Целевой уровень защищённости ИСПДн для «Медикаменте» — **УЗ-1**, так как обрабатываются
> специальные категории ПДн (сведения о здоровье) более чем 100 000 субъектов с учётом
> планируемого роста.

## Сводная таблица проблем

| #  | Категория | Зона / процесс | Проблема (As-Is) | Риск | Рекомендуемое улучшение (To-Be) |
|----|-----------|----------------|------------------|------|---------------------------------|
| 1  | Хранение | Общий диск (Excel, PDF, JPG) | Конфиденциальные данные хранятся в открытом виде в файлах на сетевом диске | Утечка при компрометации одной учётной записи AD; нарушение ст. 19 152-ФЗ | Перейти на СУБД с шифрованием at-rest; вынести персональные данные в отдельный домен с TDE |
| 2  | Хранение | Сканы (паспорта, мед.книжки, анализы) | Сканы лежат в каталогах без разграничения и шифрования | Утечка ПДн и спец. категорий | Хранилище объектов (S3-совместимое) с SSE-KMS и тегами чувствительности |
| 3  | Передача | E-mail (Exchange) | Все обмены с лабораторией, банком, поставщиками — открытым письмом | Перехват, нарушение врачебной тайны | Защищённый канал (TLS 1.3 + S/MIME), специализированный обмен с лабораторией по API |
| 4  | Передача | OLE ККМ → 1С | Локальная сеть без шифрования, без аутентификации | Перехват/подмена платежа | TLS-туннель между ККМ и приёмником, сертификатная аутентификация |
| 5  | Передача | Печать | Документы попадают в очередь принтера и могут быть забраны посторонним | Утечка медкарт | Pull-printing с аутентификацией по карте |
| 6  | Доступ | AD-группы | Доступ — «всё или ничего», нет ABAC по чувствительности данных | Сотрудник видит больше, чем нужно | RBAC + ABAC, привязка к роли и контексту (отдел, кабинет, время) |
| 7  | Доступ | Медкарта | Любой специалист может открыть карту любого пациента | Нарушение врачебной тайны (323-ФЗ) | «Break-the-glass» с обоснованием и аудитом, доступ только своим пациентам |
| 8  | Аудит | Все системы | Журналы доступа к файлам не ведутся; в 1С минимальный лог | Невозможно расследовать инцидент | Централизованный SIEM (например, Elastic + Wazuh) с подпиской на события файлового сервера и 1С |
| 9  | Целостность | Excel-журналы | Можно править задним числом без следов | Финансовые манипуляции, искажение медданных | Append-only хранилище / журналируемая СУБД с хешами событий |
| 10 | Доступность | Один локальный сервер | Нет резервирования, нет географической репликации | Простой клиники, потеря медкарт | Кластер с резервированием, регулярные бэкапы с шифрованием, тесты восстановления |
| 11 | Согласия | Анкета пациента | Формы согласия на обработку ПДн собираются на бумаге, не привязаны к системам | Невозможно технически отозвать согласие | Цифровое управление согласиями (consent management), маркировка обработки по типу согласия |
| 12 | Минимизация данных | Расширенная анкета | Собираются избыточные поля (место работы и т.д.) без цели обработки | Нарушение ст. 5 152-ФЗ, штраф РКН | Пересмотреть состав анкеты, для каждого поля указать законное основание |
| 13 | Сроки хранения | Сканы, медкарты | Хранятся бессрочно, нет процедуры удаления | Нарушение ст. 5 152-ФЗ | Политики ретеншена по классам данных, автоматическая чистка по таймеру |
| 14 | Удаление | Запросы на отзыв | Нет процесса обработки запроса субъекта на удаление | Невыполнимое требование закона | Workflow «право быть забытым»: пометка → каскадное удаление/обезличивание |
| 15 | Передача третьим лицам | Лаборатория | Нет DPA / договора об обработке ПДн с лабораторией | Совместная ответственность не оформлена | Заключить договор по ст. 6 ч. 3 152-ФЗ, перейти на API-обмен |
| 16 | Спец. категории | Диагнозы (ВИЧ, психиатрия) | Лежат вместе с обычными данными | УЗ-1 не выдерживается | Изолированный домен с дополнительным шифрованием на уровне поля (field-level encryption) |
| 17 | Бэкапы | Файловый сервер | Бэкапы (если есть) не шифруются; хранятся локально | Утечка при краже носителя | Шифрование бэкапов отдельным ключом, off-site хранение |
| 18 | Конфигурация | Все хосты | Стандартные пароли AD, отсутствие MFA | Brute-force / phishing | MFA для всех ролей с доступом к ПДн |
| 19 | Сетевой контур | LAN | Плоская сеть, нет сегментации | Латеральное перемещение злоумышленника | Сегментация на DMZ / медицинский сегмент / финансовый сегмент / админ-сегмент |
| 20 | Обработка вне РФ | Cloud-сервисы | На текущем этапе не используются, но при росте есть риск | Нарушение ст. 18 152-ФЗ (локализация) | Первичное хранение ПДн только в РФ, использовать российский ЦОД |
| 21 | Управление ключами | Шифрование | Сейчас отсутствует — ключи негде хранить | Базовая блокирующая проблема | Внедрить HSM / KMS (Yandex KMS, HashiCorp Vault) с ротацией |
| 22 | Анонимизация | BI / отчётность | Аналитик строит отчёты по «сырым» Excel | Утечка идентифицирующих данных через отчёты | Слой анонимизированной витрины для аналитики (см. Task 2) |
| 23 | Тегирование | Все данные | Нет признаков чувствительности у полей и файлов | Невозможно автоматизировать защиту | Внедрить тегирование (см. `tagging.md`) |
| 24 | Релизы / CI | Будущая разработка | Пока не реализовано — но в To-Be понадобится контроль обработки тегированных данных | Утечка из-за ошибки разработчика | DLP в CI, тесты на работу с тегами, статанализ |

## Сводка по принципам Privacy by Design

| Принцип PbD | Текущее соответствие | Что не выполнено |
|-------------|----------------------|------------------|
| 1. Проактивность, а не реактивность | ❌ | Меры реактивны (на инциденты), нет системного процесса PIA |
| 2. Privacy как default | ❌ | По умолчанию открыты все файлы для всех в AD-группе |
| 3. Встроенность в дизайн | ❌ | Нет архитектурных решений (KMS, теги, журналирование) |
| 4. Полная функциональность | ⚠️ | Часть функций (запись, оплата) уже работает, но без privacy-контролей |
| 5. Сквозная защита (end-to-end) | ❌ | Шифрование отсутствует на всех слоях |
| 6. Прозрачность | ❌ | Нет реестра обработки ПДн, нет доступной пациенту политики |
| 7. Уважение к пользователю | ❌ | Нет инструментов отзыва согласия, права на удаление |

## Что сделать в первую очередь (top-7)

1. Внедрить **KMS** (управление ключами) — фундамент для остальных мер.
2. Перенести медкарты и анкеты в **СУБД с TDE и row-level security**.
3. Включить **аудит доступа** к файловому серверу и 1С, отправлять события в SIEM.
4. Заменить e-mail с лабораторией на **API-обмен с mTLS**.
5. Внедрить **тегирование** конфиденциальных данных (см. `tagging.md`).
6. Настроить **MFA** для всех учётных записей с доступом к ПДн.
7. Запустить **процесс согласий и сроков хранения** (consent + retention).
