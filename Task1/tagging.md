# Механизм тегирования конфиденциальных данных

Тегирование — это присвоение каждому элементу данных (полю, файлу, событию)
машиночитаемых меток, которые управляют дальнейшим жизненным циклом данных:
шифрованием, маскированием, удалением, доступом и аудитом.

В «Медикаменте» тегирование решает четыре задачи:

1. Автоматически применять политики защиты (шифрование, маскирование) — без ручного
   решения разработчика.
2. Проводить DLP-контроль и аудит, понимая, какие данные пересекают границы.
3. Реализовать «право быть забытым» (152-ФЗ ст. 14): найти все объекты пациента
   и каскадно удалить/обезличить.
4. Поддерживать compliance-аудит — выгрузить реестр обрабатываемых ПДн на любой момент.

## Модель тегов

Каждый тег — это пара `<namespace>:<value>`. Используется несколько ортогональных
осей:

### 1. Ось «Класс данных» (data class)

| Тег | Значение |
|-----|----------|
| `class:pii` | Персональные данные обычные |
| `class:pii-special` | Специальные категории ПДн (здоровье, гражданство) |
| `class:medical` | Врачебная тайна (диагноз, анализы, заключения) |
| `class:medical-sensitive` | Особо чувствительные (ВИЧ, психиатрия, наркология, генетика) |
| `class:financial` | Финансовые данные, платежи |
| `class:payment-card` | Платёжные карты (PAN, CVV) |
| `class:commercial-secret` | Коммерческая тайна |
| `class:hr` | Кадровые данные сотрудников |
| `class:credentials` | Секреты, пароли, ключи |
| `class:public` | Публичная информация |

### 2. Ось «Уровень чувствительности» (sensitivity)

| Тег | Значение |
|-----|----------|
| `sensitivity:l1` | Public |
| `sensitivity:l2` | Internal |
| `sensitivity:l3` | Confidential |
| `sensitivity:l4` | Restricted |
| `sensitivity:l4-plus` | Highly Restricted |

### 3. Ось «Применяемая защита» (protection)

| Тег | Значение |
|-----|----------|
| `protect:encrypt-at-rest` | Должно быть зашифровано в покое |
| `protect:encrypt-in-transit` | Должно передаваться по TLS/mTLS |
| `protect:field-level-encrypt` | Шифрование на уровне поля (envelope) |
| `protect:tokenize` | Заменяется токеном (PAN, СНИЛС) |
| `protect:mask-in-ui` | Маскирование в UI |
| `protect:mask-in-logs` | Не писать в логи, либо хешировать |
| `protect:pseudonymize` | Псевдонимизация для аналитики |
| `protect:anonymize` | Полное обезличивание для BI/ML |

### 4. Ось «Правовое основание / lifecycle»

| Тег | Значение |
|-----|----------|
| `legal:consent-required` | Обязательно согласие субъекта |
| `legal:medical-law-323fz` | Регулируется 323-ФЗ |
| `legal:pdn-152fz` | Регулируется 152-ФЗ |
| `legal:retention:1y` ... `legal:retention:75y` | Срок хранения |
| `legal:right-to-erasure` | Подлежит удалению по запросу субъекта |
| `legal:locality:ru` | Хранится только в РФ (152-ФЗ ст. 18) |

### 5. Ось «Владелец / домен»

| Тег | Значение |
|-----|----------|
| `domain:patient` | Домен «Пациенты» |
| `domain:medical` | Домен «Медкарты / анализы» |
| `domain:payment` | Домен «Платежи» |
| `domain:hr` | Домен «Кадры» |
| `domain:inventory` | Домен «Склад / закупки» |
| `owner:<team-name>` | Команда-владелец данных |

## Где «живёт» тег

| Артефакт | Где хранится тег | Кто проставляет |
|----------|------------------|-----------------|
| Колонка БД | Комментарий колонки + реестр в каталоге данных | Дата-стюард / автоматический классификатор |
| Поле в API-контракте | OpenAPI extension `x-data-class`, `x-sensitivity` | Бэкенд-разработчик; ревью архитектора |
| Файл / объект S3 | Object tag (`Tagging`) | Сервис-загрузчик автоматически по правилам |
| Событие в Kafka | Avro/Protobuf-аннотация поля + header события | Producer-сервис |
| Лог-сообщение | Аннотация в коде через MDC / logback marker | Разработчик |
| Документ 1С | Дополнительный реквизит «Класс данных» | Бухгалтер при создании справочника |
| Файл на общем диске (legacy) | DLP-агент проставляет метку на основе содержимого | Автоматический классификатор |

## Инструменты тегирования

| Инструмент | Роль | Почему подходит |
|------------|------|-----------------|
| **Apache Atlas** | Центральный каталог данных + теги (Glossary, Classification) | Open-source, REST API, поддерживает наследование тегов через lineage |
| **OpenMetadata** | Альтернатива Atlas, проще в эксплуатации | Современный UI, встроенная PII-классификация |
| **DataHub (LinkedIn)** | Каталог + lineage + теги | Хорошая интеграция с Kafka, Spark |
| **Microsoft Purview / Yandex DataLens Catalog** | Метаданные + классификация | Если нужен managed-сервис в РФ |
| **HashiCorp Vault Transform** | Токенизация с привязкой к роли | Автоматически реализует политику для `protect:tokenize` |
| **AWS Macie / open-source Macie-like** | Автоматическое сканирование S3 и проставление тегов | Для legacy-сканов |
| **Apache Ranger / OPA / Cedar** | Применение политик доступа по тегам | Связка «тег → разрешённые роли» |
| **OpenLineage + Marquez** | Прокидывание тегов через цепочки ETL | Lineage для compliance-отчётов |
| **Logback marker + Vector + Cribl** | Фильтрация и маскирование на основе тегов в логах | Привязка тегов в коде → политика в Vector |
| **PostgreSQL `COMMENT ON COLUMN` + custom view** | Хранение тегов прямо в БД | Минимально инвазивный способ |

## Жизненный цикл тега

```
┌──────────────────────────────────────────────────────────────────────┐
│ 1. Обнаружение данных                                                │
│    Автоклассификатор (regex / ML) или дата-стюард                    │
│           ↓                                                           │
│ 2. Назначение тегов                                                  │
│    Apache Atlas / OpenMetadata + правила                             │
│           ↓                                                           │
│ 3. Применение политики                                               │
│    OPA / Vault / DLP читают тег и решают:                            │
│       • шифровать?                                                    │
│       • кому показать?                                                │
│       • можно ли записать в лог?                                      │
│           ↓                                                           │
│ 4. Lineage                                                            │
│    OpenLineage прокидывает теги через ETL/Stream → потомки           │
│           ↓                                                           │
│ 5. Аудит                                                              │
│    SIEM подписан на события «доступ к класс=L4», алерты             │
│           ↓                                                           │
│ 6. Ретеншн                                                            │
│    Retention engine читает legal:retention:* → удаляет / маскирует   │
└──────────────────────────────────────────────────────────────────────┘
```

## Пример: запись пациента в системе записей на приём

Поле `patient.diagnosis_code` получает теги:

```yaml
column: patient.diagnosis_code
tags:
  - class:medical
  - sensitivity:l4
  - protect:encrypt-at-rest
  - protect:field-level-encrypt
  - protect:mask-in-logs
  - protect:anonymize    # для BI
  - legal:medical-law-323fz
  - legal:consent-required
  - legal:retention:75y  # как мед. документация
  - legal:locality:ru
  - domain:medical
  - owner:medical-records-team
```

Что произойдёт автоматически:

- При записи в БД срабатывает trigger `field-level-encrypt`, который шифрует
  значение ключом из Vault.
- API-Gateway по тегу `sensitivity:l4` требует роль `doctor` или
  `attending_doctor=current_user`.
- Сервис логирования по `protect:mask-in-logs` подменяет значение на `[REDACTED]`.
- ETL-задача в Spark по `protect:anonymize` исключает поле из витрины BI.
- Сервис согласий блокирует обработку, если пациент отозвал согласие
  (`legal:consent-required` + статус согласия).
- Сервис ретеншена не удалит поле раньше 75 лет (медицинская документация).

## Связь с проблемами As-Is

| Проблема (см. `problems.md`) | Как тегирование её закрывает |
|-------------------------------|------------------------------|
| #6, #7 — доступ «всё или ничего» | Политики OPA на основе `sensitivity:*` и `domain:*` |
| #8 — нет аудита | SIEM-правила «алерт на чтение поля с `class:medical`» |
| #13 — бессрочное хранение | Tag `legal:retention:*` + retention engine |
| #14 — нет «права быть забытым» | Поиск по `domain:patient` + `subject_id` → каскадное удаление |
| #16 — ВИЧ вместе с обычными | Tag `class:medical-sensitive` → отдельный ключ KMS + ABAC |
| #22 — BI на «сырых» данных | Tag `protect:anonymize` обязателен для всех L4-полей в витрине |
| #23 — нет признаков чувствительности | Сам факт внедрения каталога с тегами |

## Краткий roadmap внедрения

1. **Месяц 1.** Поднять Apache Atlas + OpenMetadata, согласовать модель тегов.
2. **Месяц 2.** Описать существующие сущности (медкарта, пациент, оплата), вручную
   проставить теги — для пилота на двух процессах.
3. **Месяц 3.** Подключить OPA к API-Gateway новых сервисов, реализовать enforcement
   политик по тегам.
4. **Месяц 4.** Подключить Vault Transform для PAN, СНИЛС.
5. **Месяц 5.** Подключить автоклассификатор (regex + ML) к файловому серверу для
   разметки legacy-данных.
6. **Месяц 6.** Внедрить retention engine на основе тегов; запустить DLP в CI/CD.
