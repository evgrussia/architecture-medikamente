# Анализ текущей системы и целей миграции

## 1. Текущее состояние (As-Is)

Компания «Медикаменте» работает на смешанной полу-ручной системе:

| Компонент | Сейчас |
|-----------|--------|
| Хранение медкарт, журналов, анализов | Excel / PDF / JPG на общем диске |
| Учёт денежных средств | 1С:Бухгалтерия предприятия (файловый режим) |
| Учёт ТМЦ | 1С:Торговля и склад (файловый режим) |
| Интеграция | OLE между ККМ и 1С |
| Инфраструктура | Один физический сервер с Windows Server 2022 в офисном здании |
| Сетевые сервисы | Active Directory, DNS, DHCP, LDAP, Exchange |
| Аналитика | Bus-аналитик строит отчёты в Jupyter поверх Excel |
| Команда | 6 отделов, всего ~35 человек, IT — 3 человека |

Архитектурно это **монолит распределённого вида** — приложения разные, но
весь стек крутится на одном сервере, разделяет AD и общий диск. Любое
изменение в одном месте задевает другие; восстановление после сбоя
ограничено возможностями одного хоста.

## 2. Цели миграции

Из бизнес-постановки и Task 2:

1. **Поддержать кратный рост** (5×) при открытии филиалов.
2. **Соответствие 152-ФЗ / 323-ФЗ / ФСТЭК № 21 / ПП-1119 УЗ-1.**
3. **Защитить конфиденциальные данные** (Privacy by Design).
4. **Перейти на современную микросервисную архитектуру**, описанную в Task 2:
   API GW, Patient/EMR/Payment/Inventory/HR/Lab сервисы, отдельные домены
   данных, шина Kafka, аналитический слой с обезличиванием.
5. **Запустить мобильное приложение** и личный кабинет пациента.
6. **Реализовать MVP за пару месяцев** — самостоятельная запись пациента
   через портал, синхронизированный с ресепшеном.
7. **К концу года** — полностью автоматический процесс записи с
   уведомлениями, ABAC-доступом, аудитом и тегированием.

## 3. Компоненты, затрагиваемые миграцией

### 3.1 Бизнес-процессы

| Процесс As-Is | Куда переходит To-Be |
|---------------|----------------------|
| Запись пациента на ресепшене (Excel) | Patient Service + Appointment Service + Web/Mobile App |
| Ведение медкарты в файле Excel/PDF | EMR Service + EMR DB + Sensitive EMR DB |
| Лабораторные анализы (e-mail) | Lab Integration Service + mTLS API + токены |
| Касса (двойной ввод Excel + 1С) | Payment Service + Payment Gateway + tokenization |
| Бухгалтерия и ЗП (1С) | HR/Payroll Service (либо переход 1С в SQL-режим) |
| Учёт ТМЦ (1С Торговля) | Inventory Service (либо 1С Торговля SQL-режим) |
| BI в Jupyter поверх Excel | Anonymization Pipeline → ClickHouse Mart → DataLens / Jupyter |

### 3.2 Технологические компоненты

| Слой | Что меняется |
|------|--------------|
| **Серверное оборудование** | Один локальный сервер → кластер в РФ-ЦОДе (своя стойка или Yandex Cloud), резервный ЦОД |
| **Сеть** | Плоская LAN → сегментированная (DMZ, медицинский сегмент, финансовый, админ), VPN между филиалами, мониторинг |
| **СУБД** | Excel + 1С файлы → PostgreSQL c TDE; 1С — в клиент-серверный режим на PostgreSQL/MS SQL |
| **Хранилище файлов** | Общий диск Windows → S3-совместимое (MinIO / Yandex Object Storage) с SSE-KMS |
| **Каналы** | OLE / e-mail / SMB → HTTPS, mTLS, Kafka, JWS |
| **Идентификация** | AD-учётка → AD + ЕСИА + Keycloak + MFA |
| **Аудит** | Минимальный → SIEM (Elastic + Wazuh) |
| **Защита данных** | Нет → KMS/Vault, OPA, Consent, DLP, Retention, Data Catalog |
| **Аналитика** | Jupyter поверх Excel → Spark + ARX → ClickHouse + Feast + JupyterHub |

### 3.3 Интеграции

| Интеграция | As-Is | To-Be |
|------------|-------|-------|
| ККМ ↔ касса | OLE по локальной сети | TLS-туннель + сертификат на устройстве |

## 4. Ограничения и контекст

- **Параллельная работа As-Is и To-Be** в течение нескольких месяцев — клиника
  не может остановить приём.
- **Малая исходная команда** (3 IT-специалиста) при большом объёме изменений.
- **Жёсткий compliance** — нельзя «временно отключить» защиту ради миграции.
- **Финансовая чувствительность** — кассы и 1С формируют фискальные данные,
  их перевод требует согласованности с ОФД / ФНС.
- **Медицинская тайна** — миграция медкарт не должна разрывать историю болезни
  и юридическую преемственность документов.

## 5. Что именно подвергается риску

С точки зрения **производительности и стабильности** новой системы (это тема
задания), миграция создаёт следующие категории угроз:

1. **Производительность** — медленные ответы (latency), нехватка throughput
   при росте нагрузки, deadlocks в БД, простаивающие воркеры.
2. **Стабильность** — каскадные отказы (один микросервис уронил остальные),
   потеря сообщений, потеря данных при сбое, отказ интеграции.
3. **Целостность данных** — рассинхронизация между старым 1С / Excel и новой
   моделью, дубликаты, потерянные записи о приёме или платежах.
4. **Безопасность** — узкие места в KMS, недоступность OPA или Consent,
   что блокирует весь поток.
5. **Эксплуатация** — невозможность отладки, отсутствие compass-метрик,
   медленный rollback.

Эти классы рисков и анализируются дальше через диаграмму Исикавы
(`ishikawa.drawio` / `ishikawa.md`) по шести категориям причин.
