# Рекомендации по устранению узких мест миграции

Документ содержит конкретные **меры устранения** причин, выявленных в
[`ishikawa.md`](ishikawa.md) / [`ishikawa.drawio`](ishikawa.drawio), их
**приоритет** и **оценку трудозатрат**.

## Шкала приоритетов

| Уровень | Что значит | Срок принятия меры |
|---------|------------|---------------------|
| **P0** | Блокирующая, без неё миграцию начинать нельзя | до старта миграции |
| **P1** | Высокий риск отказа в MVP | до запуска MVP (≤ 2 мес.) |
| **P2** | Высокий риск деградации при росте | до финального состояния (≤ 12 мес.) |
| **P3** | Желательно для долгосрочной устойчивости | непрерывно |

## Шкала трудозатрат (T-shirt sizing)

`XS` — < 1 нед., `S` — 1–4 нед., `M` — 1–3 мес., `L` — 3–6 мес., `XL` — > 6 мес.

## Меры по категориям

### 🟦 Инфраструктура

| Причина (из Ishikawa) | Мера | Приоритет | Трудозатраты |
|------------------------|------|-----------|--------------|
| I1. Один локальный сервер, нет DR | Развернуть кластер k8s в РФ-ЦОДе (Yandex Cloud или своя стойка); резервный регион/ЦОД с RPO ≤ 1 ч, RTO ≤ 4 ч | **P0** | L |
| I2. Плоская сеть, нет сегментации | Сетевая сегментация: DMZ, медицинский сегмент, финансовый, админ. VLAN + Calico NetworkPolicy в k8s | **P1** | M |
| I3. Нет k8s и авто-масштабирования | HPA + Cluster Autoscaler, заранее настроенные лимиты ресурсов | **P1** | S |
| I4. HSM/KMS — единая точка отказа | Vault кластер из 3 нод + HSM-backed, на старте — KMS managed (Yandex KMS) с HA | **P0** | M |
| I5. Канал филиал ↔ ЦОД | Два независимых провайдера, шифрованный туннель (Континент/S-Terra), мониторинг RTT/loss | **P1** | M |
| I6. Нет observability | Prometheus + Grafana + Loki + Tempo (OpenTelemetry) на всех сервисах с первого дня | **P0** | M |
| I7. UPS/генератор недостаточны | Уход в облако снимает вопрос частично; для офисной инфраструктуры — увеличить мощность ДГУ + автотестирование | **P2** | S |

### 🟦 Программное обеспечение

| Причина | Мера | Приоритет | Трудозатраты |
|---------|------|-----------|--------------|
| S1. «Distributed monolith» | Domain-driven разбиение, проверка bounded contexts через event storming; на старте — несколько крупных сервисов вместо мелких | **P0** | M |
| S2. Sync REST-цепочки | Saga + event-driven через Kafka; синхронные вызовы — только короткие (один сервис в глубину) | **P1** | M |
| S3. Нет circuit breaker / retry | Resilience4j + истиo Retry-Policy с экспоненциальной задержкой и budget retry | **P1** | S |
| S4. Тяжёлые JOIN, нет индексов | Проектирование схемы под нагрузку; query plans в код-ревью; pgBadger в проде | **P1** | M |
| S5. Шифрование без кеша data-key | Envelope-encryption + кеш data-key с коротким TTL в RAM сервиса | **P1** | S |
| S6. OPA single-point | Sidecar-OPA в каждом поде + bundle-сервер; политики компилируются и кэшируются | **P1** | S |
| S7. 1С в файловом режиме | Перевод 1С в клиент-серверный режим на PostgreSQL/MS SQL до запуска MVP | **P0** | M |

### 🟦 Интеграция

| Причина | Мера | Приоритет | Трудозатраты |
|---------|------|-----------|--------------|
| N1. OLE ККМ↔1С | Промежуточный «1С-агент» в Windows-контейнере или standalone-инстансе; общение с микросервисами по REST/gRPC | **P0** | M |
| N2. Лаборатория не готова к API | На переходный период — шифрованный почтовый канал + S/MIME; параллельно вести переговоры о REST-контракте | **P1** | S (старт) / L (полный API) |
| N3. Жёсткий формат чека ОФД/ФНС | Изменения чеков только через 1С; Payment Service оборачивает 1С; обязательное тестирование с тестовым ОФД | **P1** | S |
| N4. Kafka без backpressure | Квоты на топик / клиента, ограничение по объёму и по rate; flow-control через consumer-lag метрики | **P1** | S |
| N5. Двойная запись legacy + новое | Шаблон **strangler fig**: одно из направлений становится источником истины поэтапно; CDC из старого в новое до момента переключения | **P0** | L |
| N6. ЕСИА недоступна | Запасной механизм — собственная регистрация с подтверждением SMS/e-mail; для финального состояния — обязательная ЕСИА с graceful-fallback | **P1** | S |
| N7. SMS/Push провайдер — SPOF | Два провайдера с failover (например, Smsc + MegaFon), очередь с idempotency-key | **P2** | S |

### 🟩 Данные

| Причина | Мера | Приоритет | Трудозатраты |
|---------|------|-----------|--------------|
| D1. Грязные Excel-источники | Data-cleaning пайплайн на Spark, ручная нормализация ключевых сущностей (пациенты, услуги) | **P0** | M |
| D2. Сканы без OCR | OCR (Tesseract / Yandex OCR) с верификацией оператором; не блокировать миграцию, привязывать сканы к структурированной карте | **P2** | M |
| D3. Дубликаты пациентов | MDM-сервис на старте миграции: дедупликация по ФИО+ДР+телефону, ручная верификация для конфликтов | **P0** | M |
| D4. Большой backfill | Делать backfill инкрементально, ночью, с throttling; вначале мигрировать активных пациентов (последние 12 мес.), потом архив | **P1** | M |
| D5. Нет согласий | Кампания сбора согласий до миграции; для существующих пациентов — формы при первом визите/входе в ЛК | **P0** | S (старт) / непрерывно |
| D6. Несовместимость моделей | Антикоррупционный слой (anti-corruption layer) между legacy и новыми доменами | **P1** | M |
| D7. Целостность фискальных данных | Тестовая ОФД-среда, dual-write с проверкой контрольных сумм за каждый день перед отключением старого | **P0** | S |

### 🟩 Персонал

| Причина | Мера | Приоритет | Трудозатраты |
|---------|------|-----------|--------------|
| P1. IT-отдел 3 чел. | Найм SecOps, SRE, бэк/фронт-разработчиков; усиление подрядчиками на пиковый период | **P0** | M |
| P2. Сотрудники не привычны | Программа обучения, видеоинструкции, постепенный rollout (один кабинет → филиал → сеть) | **P1** | S |
| P3. Нет DPO/SecOps | Назначить DPO (внутреннего или внешнего по аутсорс-контракту), нанять SecOps-инженера | **P0** | S |
| P4. ТД — единая точка знаний | ADR-репозиторий, knowledge sharing-сессии, дублирование критичных решений через ревью с минимум 2 архитекторами | **P1** | XS (старт) / непрерывно |
| P5. Медленный найм | Привлечение подрядной команды на старт + обучение внутренних разработчиков | **P1** | M |
| P6. Сопротивление изменениям | Программа change management: спонсорство ТД и CEO, ранние «победы» (small wins), ретроспективы | **P1** | непрерывно |
| P7. Bus-factor подрядчиков | SLA с подрядчиками, доступ к коду в репозитории компании, документация передаётся вместе с поставкой | **P1** | S |

### 🟩 Процессы

| Причина | Мера | Приоритет | Трудозатраты |
|---------|------|-----------|--------------|
| R1. Релизы без CI/CD | GitLab CI / GitHub Actions с обязательными гейтами (тесты, lint, security, IaC) | **P0** | M |
| R2. Нет нагрузочного тестирования | k6 / Gatling в pipeline; нагрузочные стенды до релиза | **P1** | S |
| R3. Нет rollback | Blue-green / canary через istio, переключение feature-flag, тестирование отката в стейджинге каждые 2 нед. | **P0** | S |
| R4. Нет on-call 24/7 | PagerDuty / Grafana OnCall + расписание, runbook'и на каждый инцидент | **P1** | S |
| R5. Big-bang миграция | **Strangler fig**: поэтапная миграция одного процесса за раз (запись → касса → склад → HR), параллельная работа As-Is и To-Be до полного переключения | **P0** | L (стратегический) |
| R6. Нет SLA/SLO/SLI | Определить SLO для критичных пользовательских journey (запись на приём, оплата); SLI собирать с прода с первого дня | **P1** | S |
| R7. Нет управления API-контрактами | Schema-registry + контрактные тесты (Pact) + semantic versioning API | **P1** | S |

## Сводная сортировка по приоритету

### P0 — без этого начинать миграцию нельзя

1. **I1.** Развернуть кластер k8s в РФ-ЦОДе с резервом.
2. **I4.** HA-KMS / Vault с HSM-backed ключами.
3. **I6.** Поднять observability стек (метрики, логи, трейсы) с первого дня.
4. **S1.** Domain-driven разбиение через event storming.
5. **S7.** Перевести 1С в клиент-серверный режим.
6. **N1.** Создать «1С-агент» для интеграции ККМ.
7. **N5.** Принять стратегию **strangler fig** для legacy.
8. **D1.** Запустить data-cleaning пайплайн.
9. **D3.** Внедрить MDM для дедупликации пациентов.
10. **D5.** Запустить сбор согласий до начала миграции.
11. **D7.** Тестовая ОФД-среда и dual-write для фискальных данных.
12. **P1.** Усилить команду наймом и подрядчиками.
13. **P3.** Назначить DPO.
14. **R1.** Настроить CI/CD с гейтами.
15. **R3.** Стратегия rollback (blue-green / canary).
16. **R5.** Поэтапная миграция (strangler fig), не big-bang.

### P1 — до запуска MVP

- I2 (сетевая сегментация), I3 (HPA + Autoscaler), I5 (резервный канал филиала).
- S2 (event-driven), S3 (CB/retry), S4 (БД-схема + индексы), S5 (кеш data-key),
  S6 (sidecar-OPA).
- N2 (S/MIME + переговоры по API лаборатории), N3 (формат чеков и тестовый
  ОФД), N4 (квоты Kafka), N6 (fallback регистрации ЕСИА).
- D4 (инкрементальный backfill), D6 (anti-corruption layer).
- P2 (обучение сотрудников), P4 (ADR + ревью), P5 (привлечение подрядчиков),
  P6 (change management), P7 (SLA подрядчиков).
- R2 (нагрузочное тестирование), R4 (on-call), R6 (SLA/SLO/SLI),
  R7 (контракты API + Pact).

### P2 — до финального состояния (через год)

- I7 (UPS/ДГУ).
- N7 (два SMS/Push-провайдера).
- D2 (OCR сканов).

### P3 — непрерывно

- Поддержание ADR, обучение сотрудников, ретроспективы, расширение DR.

## Оценка совокупной нагрузки

| Период | Кол-во мер | Условные T-shirt-баллы (XS=1, S=2, M=4, L=8, XL=16) |
|--------|------------|------------------------------------------------------|
| **P0** (до старта) | 16 | ≈ 78 баллов — порядка 6 параллельных команд на 2-3 мес. |
| **P1** (до MVP) | 22 | ≈ 70 баллов — продолжение работы команд после старта |
| **P2** (до финала) | 3 | ≈ 9 баллов |
| **P3** | непрерывно | — |

Это даёт ориентир: чтобы выдержать сроки MVP, нужно одновременно держать
4–6 параллельных стрим-команд (разработка, инфра, миграция данных, безопасность,
интеграции, бизнес-обучение).

## Итог

Самый дорогой узел — **strangler fig стратегия миграции (R5 + N5)**. Если
она не принята, все остальные меры могут «не успеть» сработать.

Самый дешёвый и одновременно важный — **наблюдаемость (I6) и CI/CD (R1)**.
Они дают возможность увидеть проблемы и быстро откатить.

Самый коварный риск — **узкие места внутри платформы безопасности**
(KMS, OPA, Consent): они становятся SPOF для всех сервисов, поэтому требуют
горизонтального масштабирования и кешей с первого дня.
