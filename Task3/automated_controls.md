# Автоматизированный контроль за данными

Документ описывает, как «Медикаменте» автоматически проверяет, что
заявленные в [`protection_tools.md`](protection_tools.md) меры действительно
работают, а нарушение правил обработки данных обнаруживается без
ручного контроля.

## Принципы

1. **Каждое правило — машинно-проверяемое**. Если правило нельзя проверить
   автоматически, оно не считается реализованным.
2. **Контроль на трёх уровнях**: design-time (CI/CD), run-time (API GW + OPA +
   DLP), post-mortem (SIEM).
3. **Контроль идёт за тегами**. Меняется тег — поведение защиты меняется
   автоматически.
4. **Закрытый цикл**. Каждое нарушение порождает алерт → инцидент в
   DPO Console → расследование → корректировка тегов/политик/кода.

## Конкретные автоматизированные проверки

### A. Design-time (CI/CD Privacy Gates)

| # | Проверка | Инструмент | Что блокируется |
|---|----------|------------|-----------------|
| A1 | Поле в OpenAPI без тега `x-sensitivity` | Spectral + custom-rules | Merge PR |
| A2 | Возврат поля `sensitivity:l4+` в публичном эндпоинте | Spectral | Merge PR |
| A3 | Schema Avro в Kafka без тегов | Apicurio + custom validator | Регистрация схемы |
| A4 | Миграция БД создаёт колонку без `COMMENT ON COLUMN` с тегами | Custom-CI step | Применение миграции |
| A5 | Жёстко зашитые секреты в коде | GitLeaks / ggshield | Push |
| A6 | log.info / log.debug с переменными, помеченными как PII | Semgrep правила | Merge PR |
| A7 | Контейнер без USER, с known CVE | Trivy | Релиз |
| A8 | Terraform создаёт PostgreSQL без TDE / RLS, S3 без SSE-KMS | OPA + tfsec | Apply |
| A9 | Отсутствие тестов на OPA-политики для нового сервиса | Conftest | Merge PR |
| A10 | Документация безопасности не обновлена для новой фичи L3+ | DocLinter | Merge PR |

### C. Post-mortem (SIEM + аудит)

| # | Правило корреляции | Инструмент | Реакция |
|---|--------------------|------------|---------|
| C1 | Один пользователь читает > 100 медкарт за час | Elastic Watcher | Алерт DPO, тикет в SOC |
| C2 | Доступ к L4 в нерабочие часы / из необычного места | Wazuh + geoIP | Алерт + блок учётки |
| C3 | Изменение OPA-политики, ослабляющее доступ | SIEM-правило | Алерт CISO, требует 4-eyes ревью |
| C4 | Изменение тегов в Data Catalog «вниз» (например, l4 → l3) | Atlas-аудит → SIEM | Алерт DPO + откат |
| C5 | Retention Engine не отработал задачу удаления | Atlas + Prometheus | Алерт SRE |

| C9 | Несанкционированная попытка чтения backup-bucket | S3 audit log | Алерт + блок IAM |
| C10 | Использование `crypto-shredding`-ключа после удаления | KMS audit | Расследование |

## Связь автоконтроля с тегами

Все правила выше **читают теги** из Data Catalog. Когда дата-стюард меняет
уровень чувствительности поля — поведение всей инфраструктуры (API GW, DLP,
SIEM, Retention) меняется без релиза кода.

Пример: поле `patient.email` было `sensitivity:l3`, риск признали выше →
повышаем до `l4` в Atlas. С этого момента:

- API GW начинает маскировать e-mail в ответах;
- логи автоматически redact-ят значение;
- DLP блокирует выгрузку этого поля без JIT-grant;
- ретеншн сокращается до новой политики L4;
- SIEM начинает алертить на массовое чтение.

## Метрики контроля (KPI)

| Метрика | Цель |
|---------|------|
| % полей в Data Catalog с проставленными тегами | ≥ 95% |
| % сервисов, использующих общий шаблон с подключёнными PbD-клиентами | 100% |
| Время от изменения тега до применения политики | < 5 минут |
| MTTD (mean-time-to-detect) утечки | < 1 час |
| MTTR (mean-time-to-respond) | < 4 часа |
| Доля релизов, заблокированных Privacy-Gate | трекать; падение указывает на регресс |
| Покрытие OPA-политик тестами | ≥ 90% |
| Время выполнения запроса субъекта на удаление | < 30 дней (по 152-ФЗ) |

## Совмещение с PbD-принципами

| Принцип PbD | Какой автоконтроль реализует |
|-------------|------------------------------|
| 1. Проактивность | A1–A10 (CI/CD-гейты), C3–C4 (контроль изменений политик) |
| 2. Default | B5–B6 (маски по умолчанию), B3 (OPA default deny) |
| 3. Embedded | A4, A8 (запрет создавать БД/таблицы без TDE и тегов) |
| 4. Full functionality | Метрики KPI отслеживают баланс защиты и удобства (не блокируем лишнего) |
| 5. End-to-end | B1, B9, C1, C9 — контроли по всему жизненному циклу |
| 6. Transparency | DPO Console + аудит SIEM, KPI публикуются |
| 7. Respect for user | KPI «время на запрос субъекта» < 30 дней |

## Итог

Автоматизированный контроль = **набор гейтов на трёх стадиях** (CI/CD, run-time,
post-mortem), работающих на основании тегов. Это превращает декларации
безопасности (политики, требования регуляторов) в **исполняемый код**: если
тег проставлен — защита есть; если защита не сработала — есть алерт; если
правило не покрыто метрикой — оно не считается реализованным.
